# AGENTS.md — Rinha de Backend 2026 (Python 3)

Fraud detection API for the Rinha de Backend 2026 challenge. Receives a transaction payload, extracts 14 scalar features, runs them through a pre-trained logistic regression model, and returns an approval decision + fraud score.

## Architecture

### Request Flow

1. Nginx LB (port 9999) round-robins to `api1:8000` / `api2:8000`
2. Falcon deserializes JSON body (`orjson` if available, stdlib `json` fallback)
3. `score_payload()` extracts 14 scalar features from the nested JSON, normalizes each to [0,1] using precomputed inverse constants
4. Logistic regression: `sigmoid(MODEL_INTERCEPT + sum(coef[i]*f[i]))` — pure Python `math.exp`, no NumPy at runtime
5. `approved = fraud_score < FRAUD_THRESHOLD` (default 0.6, env-overridable)
6. Response: `{"approved": bool, "fraud_score": float}` — always HTTP 200

### Build-Time Pipeline (`preprocess.py`)

1. Downloads `references.json.gz` + `mcc_risk.json` + `normalization.json` from the official challenge repo
2. Parses 3M reference samples, builds NumPy arrays (14D vectors, uint8 labels)
3. Trains `SGDClassifier(loss="log_loss")` with `class_weight={0:1.0, 1:1.4}` to bias toward fraud recall
4. Saves `model.json` (coef list + intercept float)
5. Deletes the large `.gz` to keep image lean

This all runs inside the Dockerfile `RUN python preprocess.py` step — no model training at runtime.

## Feature Vector (14 dimensions)

All features normalized to [0,1] with -1.0 sentinel for missing data. Order matters — it must match `MODEL_COEF` indices:

| Index | Feature | Source | Normalization |
|-------|---------|--------|---------------|
| f0 | amount | `transaction.amount` | ÷ max_amount |
| f1 | installments | `transaction.installments` | ÷ max_installments |
| f2 | amount_vs_avg | amount / `customer.avg_amount` | ÷ amount_vs_avg_ratio |
| f3 | hour_of_day | `transaction.requested_at` | ÷ 23 |
| f4 | day_of_week | derived from requested_at | ÷ 6 |
| f5 | minutes_since_last_tx | delta to `last_transaction.timestamp` | ÷ max_minutes (-1 if null) |
| f6 | km_from_last_tx | `last_transaction.km_from_current` | ÷ max_km (-1 if null) |
| f7 | km_from_home | `terminal.km_from_home` | ÷ max_km |
| f8 | tx_count_24h | `customer.tx_count_24h` | ÷ max_tx_count_24h |
| f9 | is_online | `terminal.is_online` | binary |
| f10 | card_present | `terminal.card_present` | binary |
| f11 | merchant_unknown | merchant.id ∉ customer.known_merchants | binary |
| f12 | mcc_risk | `merchant.mcc` | lookup in mcc_risk.json (default 0.5) |
| f13 | merchant_avg_amount | `merchant.avg_amount` | ÷ max_merchant_avg_amount |

Normalization constants come from `resources/normalization.json`. The app precomputes inverse constants at import time (e.g. `INV_MAX_AMOUNT = 1.0 / norm["max_amount"]`) for multiplication instead of division in the hot path.

## Gotchas & Non-Obvious Details

1. **Always return HTTP 200**: The exception handler in `FraudScoreResource.on_post` catches everything and returns `{"approved":true,"fraud_score":0.0}` with 200. HTTP errors cost -5 points in the scoring formula — never break this pattern.

2. **Docker build context is project root**: The Dockerfile is at `src/Dockerfile` but it COPY's `resources/` from the build context. You MUST build from the repo root with `-f src/Dockerfile`, not from inside `src/`. Building from `src/` fails with `"resources": not found`.

3. **No `daemon off;` in nginx.conf**: The official nginx image sets this internally; adding it causes `"daemon" directive is duplicate` errors.

4. **YAML `!reset` not supported**: Don't use Docker Compose YAML anchors with `!reset`. Use explicit service blocks instead.

5. **No NumPy at runtime**: The current implementation uses pure Python `math.exp` for sigmoid. NumPy/scikit-learn are only used in `preprocess.py` (build time). The `resources/vectors.bin` and `labels.bin` files are gitignored and no longer used at runtime — the model is `model.json` (a few KB).

6. **Precomputed inverse constants**: Feature normalization uses `INV_*` constants (multiplication) rather than dividing by max values each request. This is a deliberate micro-optimization — follow the pattern if adding new features.

7. **`_safe_ratio` returns 1e12 on zero denominator**: This sentinel propagates through `_clamp01` → 1.0, effectively marking the ratio as maximally suspicious. Don't change this without retraining the model.

8. **`_days_from_civil` is a pure-function civil calendar algorithm**: It computes days since Unix epoch without any datetime library. This avoids the overhead of `datetime.strptime` in the hot path. Don't replace it with stdlib datetime.

9. **`FRAUD_THRESHOLD` is loaded after `score_payload` is defined**: The env var check at line 161-163 overwrites the module-level `FRAUD_THRESHOLD` after the function definition but before any request. This works because Python closures resolve the name at call time, not definition time. Be careful if refactoring.

10. **orjson graceful fallback**: The import is wrapped in try/except. If orjson is missing, the app falls back to stdlib `json` with compact separators. This keeps the Docker image lean if orjson fails to install.

11. **Two Gunicorn workers**: The Dockerfile CMD uses `--workers 2`. The old AGENTS.md said 1 worker — that's outdated. Each API container gets 0.42 CPU, so 2 workers allows some parallelism under CFS quota.

## Scoring Formula (from test.js)

- **Weighted errors**: `E = fp×1 + fn×3 + errs×5` (false negatives penalized 3×, HTTP errors 5×)
- **Failure rate cutoff**: If `(fp+fn+errs)/N > 15%` → detection score = -3000 (instant disqualification)
- **p99 cutoff**: If p99 > 2000ms → latency score = -3000
- **Final score** = p99_score + detection_score (higher is better)

This means: never letting errors exceed 15% and keeping p99 under 2s are hard survival constraints.
